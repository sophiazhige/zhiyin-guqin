# 知音 · 古琴智能疗愈 —— 项目说明（评委版）

> 以两千年「五音疗疾」古训为内核，用多智能体编排 + 多模态大模型，把一床古琴变成一位随身的中医情志调理师。
>
> **一句话**：用户说出/录下此刻的状态，知音在数秒内完成「辨证 → 选音 → 开方 → 陪伴」的完整闭环，并随时间记住、读懂、回应每一位用户。

---

## 一、目标人群与商业规划

### 1.1 我们为谁而做

核心人群是**长期处于亚健康、情志失衡状态的都市人群**，他们有三个共性：症状真实存在但够不上就医、对纯西医安眠/抗焦虑药物有顾虑、对传统文化与「自然疗法」有亲近感。

| 圈层 | 典型画像 | 核心痛点 | 知音的价值 |
|------|----------|----------|------------|
| 主力人群 | 25–45 岁都市白领 | 失眠多梦、压力紧绷、肝郁情绪低落 | 睡前 10 分钟的情志疏解，不依赖药物 |
| 银发人群 | 50+ 退休/半退休 | 心悸、体质虚弱、需要陪伴感 | 温和的五脏调养 + 琴师语音陪伴 |
| 文化人群 | 国风/古琴/中医爱好者 | 想要更懂自己的个性化曲单 | 按体质五行生成的「定制琴方」 |
| 心理调节 | 有情绪困扰但抗拒「看病」标签 | 不愿进医院/心理诊所 | 去医疗化的「疗愈」入口 + 高危兜底转介 |

### 1.2 为什么是现在、为什么是古琴

- **政策与心智**：中医药、情绪健康、睡眠经济三条赛道同时升温，「非药物干预」获得越来越高的社会接受度。
- **学理根基**：《黄帝内经》「五音应五脏」——角应肝、徵应心、宫应脾、商应肺、羽应肾，知音把这套古训工程化为可计算的诊断规则，而非玄学包装。
- **差异化**：市面冥想/白噪音 App 多为「千人一曲」的内容库；知音是「**一人一辨证、一辨证一琴方**」的智能体产品，护城河在编排与数据，而非曲库本身。

### 1.3 商业模式（分层变现）

1. **订阅制（主营）**：免费体验基础辨证 + 试听；会员解锁完整疗程、个性五行报告、琴师语音畅聊、长期情绪趋势。
2. **疗程包 / 内容增值**：与古琴演奏家、中医师联名的专业疗程（失眠 21 天、疏肝 14 天等）。
3. **B 端授权**：向中医馆、康养中心、酒店/民宿、企业 EAP（员工心理援助）输出「辨证—选音」引擎与硬件方案（音响 + 屏）。
4. **数据资产（远期）**：在合规与脱敏前提下，沉淀「情志—证型—曲目—反馈」标注数据集，反哺模型与音乐治疗研究。

### 1.4 增长路径

体验钩子（说一句话即出辨证报告，可生成 3:4 国风「琴方」卡片一键分享社交媒体）→ 订阅转化（疗程与个性报告）→ 留存（长期记忆带来的「越用越懂我」）→ B 端规模化。

## 二、智能体编排（技术核心）

知音后端不是「调一次大模型」，而是一套**多智能体协作 + 规则引擎兜底 + 长短期记忆**的编排系统。下图是一次完整请求的数据流：

```
用户输入（文字 / 语音 / 实时对话）
        │
        ▼
┌─────────────────────────────────────────────┐
│  多模态接入层                                  │
│  · 浏览器 Web Speech API（前端实时语音转文字）  │
│  · StepFun ASR（后端高精度语音转写，SSE 流式）  │
│  · StepFun Realtime（wss 双向实时语音畅聊）    │
└─────────────────────────────────────────────┘
        │ transcript（统一为文本）
        ▼
┌─────────────────────────────────────────────┐
│  智能体编排层（LLMAgentService / GLM-5.1）      │
│  ① 诊断 Agent   提炼隐藏画像 + 结构化辨证        │
│  ② 推荐 Agent   仅从曲库 RAG 选曲（不许杜撰）    │
│  ③ 医师 Agent   生成诗意「回响」慰藉            │
│  ④ 琴师 Agent   实时语音对话 + 换曲推荐         │
│  ⑤ 记忆 Agent   计算五行性格底色               │
│  ⑥ 琴师 Agent   每日状态短语                   │
└─────────────────────────────────────────────┘
        │  每个 Agent 失败都回落 ↓
        ▼
┌─────────────────────────────────────────────┐
│  规则引擎兜底层（DiagnosisService）             │
│  五音五行打分 + 体质加权 + 情绪映射 + 高危检测   │
└─────────────────────────────────────────────┘
        │  读写 ↕
        ▼
┌─────────────────────────────────────────────┐
│  长短期记忆层（AgentStateRepository）           │
│  隐藏画像 / 辨证历史 / 情绪时序 / 收藏 / 播放史  │
└─────────────────────────────────────────────┘
```

### 2.1 多智能体编排（6 个职责清晰的 Agent）

每个智能体有**独立的 system prompt、独立的输入契约、强制 JSON 结构化输出**（`response_format: json_object`、`temperature: 0.2`），由 `LLMAgentService` 统一调度：

| Agent | 职责 | 输入 | 强制输出 |
|-------|------|------|----------|
| 诊断 Agent | 从自述中提炼**隐藏画像**并给出辨证 | 转写文本 + 历史画像 | 体质/证型/调理原则/音调/强度/care 信号 |
| 推荐 Agent | 从曲库中挑曲（RAG，**只能选不能造**） | 画像 + 辨证 + 完整曲库 | `track_ids`（≤4）+ 推荐理由 |
| 医师 Agent | 生成「你的回响」诗意慰藉，可对比历史变化 | 用户信息 + 最近 3 次历史 | `user_summary` + `poetic_comfort` |
| 琴师 Agent（畅聊） | 实时语音对话，按情绪即时换曲 | 当前曲目 + 画像 + 曲库 | `reply` + `new_track_ids` |
| 记忆 Agent | 计算五行性格画像（木火土金水各占比） | 全量历史画像 | 五维分值（合 100）+ 性格总结 |
| 琴师 Agent（每日） | 生成 15 字内的今日状态 | 当前画像 | `status_text` + `mood_element` |

**编排亮点**：诊断与推荐是**串行流水线**——先辨证产出画像，再把画像 + 辨证 + 曲库一起喂给推荐 Agent，保证选曲建立在辨证之上而非孤立联想。统一 Agent 入口（`/api/agent/run`）一次请求即跑完「转写 → 辨证 → 选曲」全链路。

### 2.2 多模态模型调用

| 模态 | 模型 / 通道 | 场景 |
|------|------------|------|
| 文本理解与生成 | GLM-5.1（经 Tokendance 网关） | 全部 6 个 Agent |
| 语音转文字（端侧） | 浏览器 Web Speech API | 入口页实时把心声转成文字 |
| 语音转文字（云端） | StepFun `stepaudio-2.5-asr`（SSE 流式增量） | 高精度音频上传转写 |
| 实时语音对话 | StepFun `step-audio-2` Realtime（WebSocket，PCM16 双向） | 「与琴师畅聊」语音陪伴 |
| 音频合成 | 后端按曲目五音生成预览波形（WAV） | 曲目试听指纹 |

三套语音通道**互为备份**：端侧 Web Speech 即时但精度有限，StepFun ASR 精准且可流式，二者失败再回落本地 `faster-whisper`。

### 2.3 精心标注的古琴数据集 + RAG

知音的「知识库」不是一堆音频，而是一份**人工精标的中医-古琴映射数据集**（`therapy_library.py`，单一可信源）：

- **9 大证型**（失眠多梦、肝郁气滞、脾胃虚弱、肺燥气滞、肾虚不足、心火偏旺、压力紧张、经期不适、体质虚弱），每条都标注了：所属脏腑、调理原则、对应音调、症状关键词、情绪标签、绑定曲目。
- **5 音五行档案**：角(木/肝)、徵(火/心)、宫(土/脾)、商(金/肺)、羽(水/肾)，含音色描述与「锚定曲目」。
- **35 首古琴曲目**，标注时长、调式、五行属性、疗效、来源证型（一曲可对应多证）。

**RAG 机制**：推荐 Agent 调用时，后端把**整个曲库压缩后注入上下文**（id/曲名/调式/疗效/来源证型），并在 system prompt 中硬约束「**只能从曲库里挑选**」。这是一种检索受限生成——模型负责「按画像做语义匹配与排序」，但候选集被锁死在精标曲库内，从根本上杜绝大模型「编造不存在的曲目」。

### 2.4 五行五音诊断规则引擎

这是**不依赖大模型也能独立运行**的确定性内核（`DiagnosisService`），既是主力算法之一，也是 LLM 的兜底：

- **多源加权打分**：身体症状（权重 3）、自述文本关键词（权重 2）、情绪×强度（强度/3 动态权重），命中即给对应证型累加分数。
- **体质-证型加权矩阵**：已知体质（肝郁/心火/脾虚/肺燥/肾虚）对相关证型额外 +2。
- **性别规则**：女性用户对「经期不适」证型在相关症状下 +3。
- **疗程进度感知**：连续 3 次同证型则**主动降权**，避免「永远推同一类」，引导循序调理。
- **排序与组方**：取最高分为主证，次高分达阈值则纳为兼证；按「主证曲 3 首 + 兼证 1 首 + 该音调锚定曲」组出 ≤4 首处方曲单。

### 2.5 兜底与异常处理（工程可靠性）

知音在**每一层都做了优雅降级**，保证「大模型挂了产品也不挂」：

- **三级兜底链**：LLM Agent 失败 → 规则引擎辨证 → 静态诗意文案，用户永远拿到结果。
- **多端点重试**：LLM 网关依次尝试 3 个候选 endpoint，任一成功即返回。
- **输出安全护栏**（`_guard_output`）：拦截自杀/自残等高危内容，命中即丢弃该次模型输出并回退。
- **高危情志检测**：自述中出现「不想活/撑不住了」等词，辨证状态置为 `care`，**不进入音乐流程**，改为引导联系心理援助——把安全置于体验之上。
- **多模态降级**：StepFun ASR 失败自动回落本地 faster-whisper；实时语音畅聊在前端把 WebSocket 错误码、断连原因显式回显，便于定位。
- **数据持久化安全**：记忆文件采用 `fcntl` 文件锁 + 临时文件原子替换写入，读失败返回空态而非崩溃。
- **前端容错**：辨证/回响接口失败时，前端用用户输入动态生成半文言兜底文案，页面不空白。

### 2.6 长短期记忆

知音「越用越懂你」的能力来自分层记忆（`AgentStateRepository`）：

- **长期记忆 —— 隐藏画像**：体质、长期性格特质、慢性症状、风险标记、偏好音调。每次辨证由诊断 Agent 增量更新并**去重合并**，沉淀成跨会话的用户模型。
- **短期记忆 —— 时序日志**：辨证历史（保留最近 30 条）、情绪/证型时序（最近 60 条，用于趋势分析）、播放历史（最近 100 条）。
- **记忆的闭环消费**：医师 Agent 写「回响」时对比最近 3 次历史→「比上次轻松了些」；规则引擎读慢性症状与体质做加权；记忆 Agent 据全量历史算五行性格底色。记忆不是存了就放着，而是真正回流进每一次决策。

### 2.7 工具调用

智能体之上挂载了一组**可被编排调用的工具/接口**，形成完整能力面（共 20 个 API）：

- **检索类**：曲库查询、单曲详情、按辨证组方。
- **生成类**：辨证（`/api/diagnosis`）、统一 Agent（`/api/agent/run`）、回响（`/api/resonance`）、每日状态、五行性格、琴师畅聊。
- **记忆/状态类**：用户画像建立与更新、收藏增删查、播放历史读写。
- **多媒体类**：音频转写、曲目波形预览合成。

推荐 Agent 的「从曲库选曲」本质即一次**受约束的工具调用**：工具（曲库）定义了可选动作空间，模型在其中决策，结果再被后端校验（过滤不存在的 track_id）后才生效。

## 三、前后端技术栈

### 3.1 前端

| 维度 | 选型 |
|------|------|
| 框架 | React 18 + TypeScript |
| 构建 | Vite 6 |
| 样式 | Tailwind CSS v4 |
| 图标 | lucide-react |
| 状态管理 | React Context（`AudioProvider` 全局共享播放、`IntakeProvider` 录入状态）|
| 语音输入 | 浏览器原生 Web Speech API |
| 实时语音 | WebSocket + Web Audio API（PCM16 编码 / 播放）|
| 分享卡片 | Canvas 动态绘制 3:4 国风「琴方」图，一键保存分享 |
| 交互动效 | 自研翻页动画、五行旋转辨证盘、呼吸/水波 SVG 动效 |

**前端工程亮点**：跨页面共享音频上下文——从收藏夹/历史/疗程任意入口播放都不会打断当前播放器，实现「全局一个播放器」的连续聆听体验。

### 3.2 后端

| 维度 | 选型 |
|------|------|
| 框架 | Python FastAPI + Pydantic |
| 服务编排 | 自研 `LLMAgentService`（多 Agent 调度）+ `DiagnosisService`（规则引擎）|
| LLM 接入 | GLM-5.1（Tokendance 网关，OpenAI 兼容协议）|
| 语音能力 | StepFun ASR / Realtime；本地 faster-whisper 兜底 |
| 持久化 | JSON 文件存储 + `fcntl` 文件锁 + 原子写入（轻量、零外部依赖）|
| 配置 | 环境变量驱动（`Settings` dataclass），密钥不入库 |
| 网络层 | 标准库 `urllib`（无重依赖）+ SSE 流式解析 |
| 跨域 | CORS 中间件 |

**架构理念**：核心诊断逻辑与大模型**解耦**——规则引擎可独立产出专业辨证结果，大模型负责锦上添花（语义理解、个性表达、画像提炼）。这让产品在「无网络/无密钥/模型故障」下仍可用，也便于后续把规则沉淀为可解释的医学审计依据。

## 四、补充亮点

### 4.1 产品完整度
不是 Demo，而是覆盖**完整用户旅程**的可用产品：欢迎引导 → 多卡片录入（症状/情绪/强度/睡眠/自述）→ 五行辨证盘 → 辨证结果 → 诗意回响 → 琴方处方笺 → 沉浸播放 → 疗程进度 → 收藏/历史 → 个人五行画像 → 琴师语音畅聊。

### 4.2 设计与文化表达
通篇国风视觉（宣纸纹理、印章、毛笔字体、五行配色），把「中医辨证」这种偏专业的内容做成了**可感知、愿分享**的轻疗愈体验，降低了传统文化产品的认知门槛。

### 4.3 安全与伦理优先
高危情绪不进入音乐流程而是转介心理援助；LLM 输出经安全护栏过滤——在「情志健康」这一敏感领域，把用户安全放在产品指标之前。

### 4.4 可解释性
每次辨证都附带 `rationale`（最多 3 条中文短句）说明「为什么这样判、为什么推这些曲」，而非黑箱输出——这对医疗健康类产品的信任建立至关重要。

### 4.5 可扩展性
证型/曲目/规则集中在单一数据源（`therapy_library.py`），新增证型或曲目只需改一处；Agent 以 system prompt 解耦，新增智能体（如「营养/起居建议 Agent」）成本极低。

### 5.3 重点验证了什么

- **高风险兜底路径稳定**：8 个高危触发词（自杀/轻生/不想活…）经 TC-020、TC-021、TC-045、TC-052 多入口验证，均能正确将状态置为 `care`、填充危机提示并暂停音乐流程。
- **异常输入全覆盖**：超范围 intensity（Pydantic 422 拦截）、缺失必填字段、空白/超长（≥500 字）/Emoji/英文文本、重复收藏去重、删除不存在条目静默忽略、无效曲目 404——逐一验证无崩溃。
- **记忆健壮性**：诊断历史 30 条滚动窗口（TC-100）、列表字段自动去重（TC-099）、新用户自动初始化均通过。
- **LLM 全链路降级**：所有依赖大模型的接口在「未配置/抛异常」时都回落规则引擎或模板文案，测试在无 LLM 密钥环境下依然 100% 通过——印证了「模型挂了产品不挂」的解耦设计。

### 5.4 发现并修复的真实 Bug

| | 内容 |
|--|------|
| 用例 | TC-058 |
| 标题 | 用户画像变更检测字段名不一致 |
| 现象 | 相同生辰数据重复提交时，`profile_changed` 始终误返回 `true`，触发冗余画像写入 |
| 根因 | `update_user_profile` 用 payload 字段名 `name` 直接查 `hidden_profile`，而存储时被映射为 `name_hint` |
| 修复 | 引入 `storage_key` 映射表，比较时用真实存储键（`main.py` 第 349–351 行）|

这正体现了测试的价值——一个隐蔽的字段映射 Bug 在上线前被自动化用例捕获并修复。
